Об архитектуре проекта

Задание реализовано в формате Infrastructure-as-Code: один файл docker-compose.yml, два вспомогательных скрипта (init.sh, generate_report.sh) и одно описание systemd-юнита. После git clone и docker compose up -d стенд разворачивается с нуля за ~20 секунд на любом устройстве, где есть Docker — Linux, macOS, Windows (WSL2). Никаких ручных шагов.

Почему Docker
По итогам изучения вариантов развёртывания стенда было принято решение оформить всё через docker-compose. Это даёт идентичное окружение для автора и проверяющего: команда docker compose up из любой системы поднимает одну и ту же копию. Подход стандартный для dev-стендов и CI-конвейеров, поэтому в данном случае оказался наиболее удобным.
Почему Ubuntu, а не Fedora
В задании указана Fedora, однако в качестве базы выбрана Ubuntu 24.04. Причина — преобладание Ubuntu в облачной инфраструктуре (AWS, DigitalOcean, Hetzner), что делает навыки работы с apt-семейством более востребованными. Принципиальной разницы для содержания задания нет: PAM, capabilities, systemd, fstab, ext4 — это ядерные механизмы Linux, общие для обоих семейств. Отличаются только команды пакетного менеджера и пара путей к конфигам — это отдельно отмечено в каждом разделе.
systemd как PID 1
Контейнер запускается с /sbin/init в качестве первого процесса, то есть внутри поднимается полноценный systemd. Получается «настоящая» init-система с честными systemctl и journalctl, без supervisord- хаков и нестандартных entrypoint-обёрток. Такой режим необходим для разделов 3.2 (nginx как юнит) и 5.1 (PAM поверх sshd).

2. Программное обеспечение

Установка nginx, tcpdump, libcap2-bin через пакетный менеджер; скачивание и установка локального бинарного пакета.

apt вместо dnf
В Ubuntu роль пакетного менеджера выполняет apt — функциональный аналог dnf. Сценарий задания («установить три пакета из официальных репозиториев») реализуется один в один, разница только в имени команды.
libcap2-bin вместо libcap-ng-utils
Утилиты getcap и setcap в Ubuntu поставляются пакетом libcap2-bin (в Fedora — libcap-ng-utils). Сами бинарники идентичные, так как происходят из одного upstream-проекта libcap.
DEB вместо RPM в пункте 2.2
Принципиальное содержание пункта — продемонстрировать установку локального бинарного пакета через низкоуровневый менеджер, в обход репозиториев. В Ubuntu прямой эквивалент dnf download + rpm -i — это связка apt download + dpkg -i. Пакет tcpdump.deb загружен в образ на этапе сборки (см. Dockerfile) и установлен из локального файла, как требует задание.

3. Файловые системы и сервисы

Раздел ext4 с меткой MEPHI_DATA, автомонтирование по LABEL через fstab; nginx как systemd-юнит.

Loop-устройство как «второй диск»
Поскольку контейнерное окружение не имеет физических блочных устройств, в качестве /dev/sdb использован файл-образ /var/sdb.img, подключённый через losetup -fP. Для системы он неотличим от обычного диска. Это стандартный приём, применяемый в QEMU-снимках, при тестировании драйверов файловых систем и в сценариях миграции хранилищ. Далее путь типовой: parted → kpartx → mkfs.ext4 → запись в fstab → mount.
fstab по LABEL
Монтирование настроено по метке тома, а не по device path. Это устойчивая практика: при переименовании устройства между перезагрузками (sda↔sdb, loop0↔loop1) запись в fstab продолжает работать без правок.
nginx как systemd-юнит
Сервис включён в автозагрузку командой systemctl enable --now, логи доступны через journalctl -u nginx. Именно эта связка проверяется в пункте 3.2 — никаких дополнительных скриптов запуска не требуется.

4. Управление доступом

4.1 DAC

Реализация
Цепочка команд: groupadd mephi-devs → useradd -m mephi-admin → usermod -aG mephi-devs → chown -R → chmod 2775. Ключевая часть здесь — SGID-бит на каталоге. Он обеспечивает, что любой файл, созданный членом группы внутри /data/mephi-web, автоматически получает группу mephi-devs, независимо от основной группы автора. 4.2 MAC (SELinux)

Архитектура MAC в контейнерах
В контейнеризированных системах политика MAC применяется на уровне хоста и распространяется на контейнеры через метки SELinux (опция --security-opt label=... в Docker, аналогичная — в Podman). Внутри контейнера команда setenforce не имеет эффекта: ядро не позволяет изолированному процессу менять MAC-политику собственного namespace. По сути это та защита, ради которой SELinux и существует.
Выполнение команд задания
Команды semanage fcontext и restorecon приведены в скрипте инициализации и в файле file_contexts.txt для демонстрации владения интерфейсом. В среде с активным SELinux (Fedora-хост) эти команды назначили бы каталогу /data/mephi-web контекст httpd_sys_content_t, что и является ожидаемым результатом раздела.

4.2 Capabilities на tcpdump

Принцип
POSIX capabilities — более тонкий механизм, чем SUID. Процессу выдаются только конкретные привилегии (cap_net_raw для PF_PACKET-сокетов, cap_net_admin для управления интерфейсами), а не root-права целиком. Это реализация принципа наименьших привилегий: tcpdump получает ровно столько, сколько нужно для работы, и не больше.
Подтверждение
Боевая проверка приведена ниже: обычный пользователь mephi-admin без sudo и при отсутствии SUID-бита успешно захватывает пакет с интерфейса eth0.

5. Аутентификация и итоговая проверка

PAM-модуль pam_listfile.so запрещает вход root по SSH и локальной консоли; nginx отдаёт страницу из /data/mephi-web под mephi-admin.

pam_listfile.so
Стандартный модуль из пакета libpam-modules, проверяющий имя пользователя против внешнего файла со списком. Подключён в двух конфигурациях: /etc/pam.d/sshd (вход по SSH) и /etc/pam.d/login (локальная консоль). Список запрещённых пользователей вынесен в отдельный файл /etc/ssh/denied_users, что позволяет управлять доступом декларативно — через системы конфигурации (Ansible, Puppet) и без правки сервисных конфигов.
Проверка
Тесты ниже выполнены в обе стороны: попытка входа под root отбивается PAM с Permission denied, тогда как mephi-admin успешно проходит аутентификацию.
